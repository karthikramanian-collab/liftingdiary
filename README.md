# LiftingDiary

A personal workout tracking app for serious lifters. Log every set, track progressive overload, and build a permanent record of your training — one session at a time.

---

## What it does

- Log workouts with exercises, sets, reps, weight, and RPE
- Track warmup vs working sets
- View training history and a workout calendar heatmap
- Switch between kg / lbs — preference persisted via cookie
- Industrial-themed UI built for focus, not fluff

## Architecture

The project is an **npm workspace monorepo** with two packages:

```
liftingdiary/
├── backend/                    # @liftingdiary/backend — DB layer, no framework deps
│   ├── src/db/
│   │   ├── schema.ts           # Drizzle table definitions + relations
│   │   └── index.ts            # postgres + drizzle client (exported as package root)
│   ├── drizzle/                # Migration SQL files (generated by drizzle-kit)
│   └── drizzle.config.ts
│
└── frontend/                   # @liftingdiary/frontend — Next.js app
    └── src/
        ├── app/
        │   ├── _components/            # Shared layout components (NavAuth, PageCTAs)
        │   ├── layout.tsx              # Root layout — ClerkProvider, nav, fonts
        │   ├── page.tsx                # Landing page
        │   ├── dashboard/
        │   │   ├── page.tsx            # Training history + calendar
        │   │   ├── layout.tsx          # Dashboard shell (auth-protected)
        │   │   ├── _components/
        │   │   │   ├── DashboardClient.tsx   # Unit toggle + view switcher
        │   │   │   ├── WorkoutCalendar.tsx   # Heatmap calendar of sessions
        │   │   │   └── WorkoutList.tsx       # List of past workouts
        │   │   └── workout/
        │   │       ├── new/
        │   │       │   ├── page.tsx                  # New session form page
        │   │       │   └── _components/
        │   │       │       └── NewWorkoutForm.tsx     # Date/name form → createWorkout
        │   │       └── [id]/
        │   │           ├── page.tsx                  # Workout detail page
        │   │           └── _components/
        │   │               ├── WorkoutView.tsx        # Full session view + controls
        │   │               ├── ExerciseCard.tsx       # Per-exercise set logging UI
        │   │               └── AddExerciseModal.tsx   # Exercise picker modal
        ├── actions/                    # Next.js Server Actions (auth-gated writes)
        │   ├── workout-actions.ts      # createWorkout, finishWorkout, updateWorkoutName, deleteWorkout
        │   ├── workout-exercise-actions.ts  # addExerciseToWorkout, removeExerciseFromWorkout
        │   ├── set-actions.ts          # logSet, deleteSet
        │   └── preference-actions.ts  # setUnitPreference (kg/lbs cookie)
        ├── data/                       # Server-only data fetchers (no client bundle)
        │   ├── workouts.ts             # getWorkoutSummaries, getWorkoutDetail
        │   └── exercises.ts            # getExercises
        ├── lib/
        │   ├── units.ts                # displayWeight, toKg, toDateKey
        │   └── utils.ts                # clsx/tailwind-merge helper
        ├── types/
        │   └── workout.ts              # WorkoutSummary, WorkoutDetail, SetDetail, ExerciseOption, Unit
        └── middleware.ts               # Clerk auth middleware (protects /dashboard)
```

### Key design decisions

- **Backend as a workspace package** — `@liftingdiary/backend` is imported by the frontend via `"@liftingdiary/backend": "*"`. This keeps the DB layer decoupled and typechecked independently.
- **Server Actions for all writes** — every mutation goes through a `'use server'` action. Each action re-verifies ownership via `auth()` before touching the DB.
- **`data/` layer marked `server-only`** — prevents accidental import of DB queries into client components.
- **Unit preference via cookie** — kg/lbs choice is written server-side and read at page render, so the correct unit is rendered without a client flash.

## Database Schema

Four tables with cascade deletes on the workout → sets path:

```
exercises
  id, name, muscle_group, category, created_by, created_at

workouts
  id, user_id (Clerk), name, notes, started_at, finished_at, created_at
  index: workouts_user_id_idx

workout_exercises  (junction: workout ↔ exercise)
  id, workout_id → workouts (cascade), exercise_id → exercises, order_index, notes, created_at

sets
  id, workout_exercise_id → workout_exercises (cascade)
  set_number, reps, weight_kg (numeric 8,3), duration_seconds, rpe (numeric 3,1)
  is_warmup, notes, created_at
```

Drizzle relations are defined in `backend/src/db/schema.ts` for typed relational queries.

## Getting Started

### Prerequisites

- Node.js 20+
- A [Supabase](https://supabase.com) project (PostgreSQL)
- A [Clerk](https://clerk.com) application

### Environment variables

Copy `.env.example` to `.env` and fill in:

```bash
DATABASE_URL=postgresql://...   # Supabase connection string
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_...
CLERK_SECRET_KEY=sk_...
```

### Install and run

```bash
npm install       # installs all workspace deps
npm run dev       # starts Next.js dev server (frontend workspace)
```

Open [http://localhost:3000](http://localhost:3000).

### Database migrations

```bash
# Generate a migration from schema changes
npm run db:generate

# Push migrations to the database
npm run db:migrate

# Or push schema directly (dev only)
npm run db:push
```
